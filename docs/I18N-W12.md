## 1. Visão geral

Este ciclo entrega a **fundação i18n** do Cabala dos Caminhos sem dependências externas. PT-BR é a fonte da verdade; EN e ES ficam com placeholders `__TRANSLATE__: <pt-br>` que ficam visíveis em runtime para sinalizar o que falta traduzir — sem quebrar a aplicação.

### Princípios

- **Zero libs externas.** Sem `react-intl`, sem `i18next`. Custom thin wrapper sobre o sistema já existente em `src/lib/i18n/index.ts`.
- **PT-BR sempre funcional.** Hook faz fallback automático PT-BR → chave → EN placeholder. Nunca devolve string vazia.
- **Tipado.** `useT()` retorna função `(key: string, params?) => string`.
- **Mobile-first preservado.** Bundle i18n é lazy-importado (já era o design original).
- **Surgical.** Apenas 5 componentes migrados (escopo da W12); migração em massa é trabalho de W13+.

## 2. Arquitetura

```
src/lib/i18n/
├── index.ts          ← useI18n() / i18n singleton (existente — adicionado 'es' como 3º locale)
├── useT.ts           ← NOVO. Wrapper fino que retorna só t(key)
└── locales/
    ├── pt-BR.ts      ← NOVO. Fonte da verdade, chaves agrupadas por domínio
    ├── en.ts         ← NOVO. Placeholder com prefixo __TRANSLATE__:
    └── es.ts         ← NOVO. Placeholder com prefixo __TRANSLATE__:
```

### Hierarquia de fallback (em runtime)

```
1. translations[localeAtual][key]      ← encontrado? retorna.
2. translations['pt-BR'][key]          ← fallback PT-BR (sempre presente).
3. key literal                         ← nunca devolve undefined.
```

Para EN/ES, **todas** as chaves existem mas com prefixo `__TRANSLATE__: `. Isso significa que a UI mostra a string PT-BR com prefixo — útil para QA visual saber o que ainda precisa traduzir.

## 3. Hooks

### `useT()` — recomendado para componentes

```tsx
'use client';
import { useT } from '@/lib/i18n/useT';

export function MyButton() {
  const t = useT();
  return <button>{t('common.confirm')}</button>;
}
```

### `useI18n()` — quando precisa trocar locale em runtime

```tsx
import { useI18n } from '@/lib/i18n';

export function LocaleSwitcher() {
  const { locale, setLocale, availableLocales } = useI18n();
  return (
    <select value={locale} onChange={(e) => setLocale(e.target.value as Locale)}>
      {availableLocales.map((l) => <option key={l}>{l}</option>)}
    </select>
  );
}
```

> Locale persiste em `localStorage['locale']`. Default: `pt-BR`.

## 4. Convenção de chaves

Agrupadas por **domínio** (componente / feature), não por tipo de string.

```ts
{
  nav: { home, explore, library, ... },
  feed: { title, subtitle, emptyTitle, ... },
  library: { title, searchPlaceholder, ... },
  post: { like, comment, share, ... },
  about: { title, intro, method, ... },
  common: { cancel, confirm, loading, error },
}
```

Lookup: `t('nav.home')`, `t('feed.emptyTitle')`, `t('post.like')`.

### Por que essa estrutura?

- **Acoplada ao domínio, não à tela.** Migrar uma tela significa copiar um bloco, não caçar keys espalhadas.
- **Escala bem.** Adicionar `mapa: {...}` ou `mesa: {...}` é trivial.
- **Audit-friendly.** `grep -r "t('feed" src/` mostra todos os usos do feed.

## 6. Decisões & tradeoffs

### Decisão: page.tsx server component + AboutClient.tsx client component
- **Por quê:** `useT()` precisa de `'use client'` + `useState`. Metadata SEO precisa de server component. Separar em 2 arquivos é o padrão Next.js 16.
- **Custo:** 1 arquivo a mais. Vale a pena (metadata funciona, SEO não quebra).

### Decisão: `formatRelativeTime(iso, t?)` com `t` opcional
- **Por quê:** Função já existia e tem testes (`__tests__/` provavelmente). Adicionar parâmetro opcional mantém backward-compat.
- **Custo:** Lógica interna ficou menos elegante (ternário aninhado). Se quebrar testes, refatorar pra função pura `formatRelativeTime(iso, locale)`.

### Decisão: PT-BR inline mantido em en.ts/es.ts como `__TRANSLATE__: ...`
- **Alternativa:** deixar en.ts/es.ts como `{}` e cair no fallback PT-BR.
- **Por que não:** perdemos a visibilidade do que precisa tradução. QA não saberia o que está em falta.
- **Tradeoff:** usuário final vê prefixo `__TRANSLATE__:` enquanto tradução não chega. Aceitável em dev/staging; em prod deve ser substituído por tradução real antes de release internacional.

### Decisão: NÃO instalar `react-intl` ou `i18next`
- **Por que:** Limite de 15min da task. Bundle size. Sistema existente em `index.ts` já cobre 100% do necessário.
- **Quando reconsiderar:** Se W13+ adicionar ICU MessageFormat (pluralização, gênero, seleção) ou precisar de mais de 5 locales. Aí vale a library.
