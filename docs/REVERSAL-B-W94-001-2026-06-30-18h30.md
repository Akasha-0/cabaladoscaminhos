## Root cause of the false positive

The audit-before-claim rule (introduced 17:01 UTC after B-W94-001 detection) was correctly established as policy, but its **implementation was missing the `git fetch origin` prerequisite step**.

The previous auditors (sessions 414867512484112, 414874845585504, 414882221191338) ran:

```bash
git rev-parse --verify f28ef5ef^{commit}  →  unknown revision (INVALID)
```

This `unknown revision` was a **shallow-clone artifact**. The sandbox `git clone` defaulted to `--depth=1` (or similar shallow mode), which means only `main`'s HEAD commit is in local objects. Branch refs (`refs/heads/w94/*`) and their commits were NOT auto-fetched.

The correct audit flow (now documented in B-W94-002):

```bash
# Step 1: server-side ref check (no local objects needed)
git ls-remote origin 'refs/heads/w##/*'

# Step 2: if ls-remote shows refs, fetch them
git fetch origin 'refs/heads/w##/*:refs/remotes/origin/w##/*'

# Step 3: local object check (now valid)
git rev-parse --verify <SHA>^{commit}

# Step 4: if rev-parse still fails despite fetch, unshallow
git fetch --unshallow origin
git rev-parse --verify <SHA>^{commit}
```

`ls-remote` hits the server-side ref database directly and is the canonical pre-rev-parse check.
