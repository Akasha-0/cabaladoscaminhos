### 2. A Recomendação Arquitetônica: O Modelo Híbrido (Core + Data Layer)

A melhor tecnologia não é uma só, mas a sobreposição estratégica de duas camadas. Para que o Akasha seja lido como uma tecnologia superior e mantenha uma fluidez estética impecável, a arquitetura ideal do frontend se divide em:

**Camada de Fundo (A Atmosfera): WebGL / Three.js Leve**

No fundo da tela, rodando de forma assíncrona, um shader simples em WebGL (ou React Three Fiber) renderiza um Toroide etéreo girando lentamente em wireframe brilhante ou fumaça de partículas. Ele não carrega dados, apenas define a frequência vibracional da interface.

**Camada de Dados (A Mandala Operacional): SVG + D3.js (Lógica) + Glassmorphism**

Sobre o fundo 3D, a mandala interativa real é renderizada em **SVG bidimensional**.

- **A Geometria:** Você usa os motores matemáticos do **D3.js** _apenas sob o capô_ para calcular as coordenadas polares exatas das 12 casas astrológicas, dos 11 nós tântricos e do centro.
    
- **A Renderização:** Você injeta essas coordenadas em elementos **SVG**. O SVG permite que cada nó, linha ou planeta seja um elemento do DOM clicável e nítido em qualquer tela.
    
- **A Estética:** Você aplica filtros CSS (backdrop-filter, drop-shadow, gradientes radiantes) nos SVGs para dar o efeito de _Glassmorphism_.

## 1. O Que Descartar e Por Quê

- **Weaviate / Pinecone (Cloud):** Devem ser descartados. ## 2. A Arquitetura Definitiva: Markdown (Origem) + pgvector (Operação)

A união do estilo de escrita em Markdown com o PostgreSQL cria um sistema imbatível de **RAG (Retrieval-Augmented Generation) Híbrido**. Ele separa o ambiente onde o humano _escreve_ do ambiente onde a máquina _pesquisa_.

### Camada A: O Santuário do Conhecimento (Markdown + YAML)

A sua base de dados não deve ser editada diretamente em tabelas frias de SQL. Manter os textos sagrados, lendas dos Orixás, rituais e aspectos astrológicos em formato Markdown com cabeçalhos YAML (Frontmatter) é a forma mais limpa e elegante de organizar o conteúdo.

- **A Vantagem:** Você escreve e versiona a sabedoria no repositório de código. O Markdown permite uma formatação rica e legível para você (tabelas, listas, ênfases), enquanto o YAML no topo do arquivo define as tags matemáticas exatas (Elementos, Odus, Signos).
    

### Camada B: O Motor de Busca (PostgreSQL + pgvector + JSONB)

Como o PostgreSQL já faz parte da sua infraestrutura, ele será o único banco de dados necessário. Você utilizará a extensão `pgvector` para armazenar a semântica e colunas `JSONB` para cruzar as correspondências.

- **O Pipeline de Ingestão:** Toda vez que você atualiza o repositório de arquivos Markdown, um script simples (Node.js/Python) lê o YAML, extrai o texto, gera o _embedding_ (o vetor matemático do texto) e salva tudo no PostgreSQL.
    
- **A Tabela "Grimorio":**
    
    - Coluna 1: `id`
        
    - Coluna 2: `categoria` (Ex: "Erva", "Transito_Astrologico")
        
    - Coluna 3: `metadata` (JSONB contendo as tags: signos, odus, etc)
        
    - Coluna 4: `conteudo` (O texto Markdown puro)
        
    - Coluna 5: `embedding` (O vetor gerado pelo pgvector)
        

## 3. Como Isso Impede a Alucinação da IA?

Quando o usuário precisa de um diagnóstico, o sistema faz uma **Busca Híbrida** no banco de dados antes de chamar o Agente IA:

1. **Filtro Determinístico (A Regra de Ouro):** O sistema usa uma query SQL comum na coluna JSONB para buscar _apenas_ os arquivos que correspondem matematicamente ao usuário. (Ex: `SELECT * FROM grimorio WHERE metadata->>'elemento' = 'Agua'`).
    
2. **Filtro Semântico (pgvector):** Dentre os resultados do passo 1, o `pgvector` encontra os textos que mais se aproximam da dor atual do usuário (baseado no quiz do onboarding ou trânsito do dia).
    
3. **Injeção de Contexto:** O sistema injeta o texto exato do Markdown no _System Prompt_ do Agente com a instrução rígida: _"Utilize APENAS os arquétipos e receitas abaixo para formular sua resposta."_
    

Com essa arquitetura, a IA se torna apenas um "sintetizador poético" de uma verdade que você já validou e que está armazenada e indexada no seu próprio banco de dados.

Pensando no script que fará a leitura desses arquivos Markdown e a geração dos vetores matemáticos para o PostgreSQL, você planeja usar um modelo de _embeddings_ hospedado externamente (via APIs como as disponíveis no OpenRouter) ou prefere rodar um modelo open-source leve (como o `all-MiniLM-L6-v2`) diretamente no terminal da sua máquina para não gerar custos por token nessa indexação?

A escolha ideal aqui é a **Opção 3 (nomic-embed-text via Ollama local)**, mas existe uma ressalva importante que precisamos desmistificar sobre os custos da nuvem para que você tome a decisão com o cenário completo.

### 1. A Desmistificação do Custo (A Verdade sobre Embeddings)

Gerar texto com a IA (o raciocínio do Agente Oracular) é o que consome dinheiro real. Gerar _embeddings_ é quase de graça. O modelo `text-embedding-3-small` da OpenAI, por exemplo, custa cerca de US$ 0,02 por 1 milhão de tokens.

Você poderia vetorizar todo o Grimório — todos os Odus, a Árvore da Vida, as ervas e os 11 Corpos Tântricos — centenas de vezes e não gastaria sequer 1 dólar. Portanto, o receio de "explodir custos" com a vetorização da base de dados é um mito.

### 2. O Caminho da Soberania (A Vitória da Opção 3)

Apesar de a nuvem ser barata, a **Opção 3** é a jogada de mestre para a arquitetura do Akasha. Rodar o Ollama e gerenciar os motores localmente no terminal de um servidor privado Linux garante o controle absoluto do ecossistema e total soberania dos dados.

- **Privacidade Absoluta:** O seu conhecimento espiritual não passeia por APIs externas para ser indexado. O tráfego de dados acontece internamente, direto do seu repositório para o seu banco PostgreSQL (`pgvector`).
    
- **Janela de Contexto Brutal:** O `nomic-embed-text` foi treinado especificamente para lidar com até 8.192 tokens de contexto, o que o torna incrivelmente eficiente para arquivos Markdown densos, como itans profundos ou regras astrológicas complexas.
    
- **Latência Zero:** O script de ingestão chama a porta local (`localhost:11434`) e o vetor é gerado e salvo instantaneamente.
    

### 3. Por que descartar a Opção 4?

O `all-MiniLM-L6-v2` via HuggingFace Inference API impõe limites de requisição (_rate limits_) e adiciona latência de rede. Além disso, por ser um modelo muito leve e mais antigo, ele tem um desempenho inferior ao capturar as nuances e a semântica complexa da língua portuguesa (pt-BR) quando comparado ao Nomic ou aos modelos da OpenAI/Gemini.

### 1. Por que descartar a Opção 1 (GitHub Actions pura)?

O instinto natural de todo engenheiro moderno é jogar a automação no GitHub Actions. Porém, no cenário do Sistema Akasha, isso é uma **armadilha de segurança e arquitetura**. Se o GitHub Actions for responsável por gerar os vetores, ele precisaria de duas coisas:

1. Ter acesso direto ao seu banco de dados PostgreSQL de produção (expondo portas para a internet).
    
2. Ter acesso à sua instância local do Ollama para gerar os _embeddings_, o que seria impossível sem expor a API da sua IA privada para o mundo exterior.
    

### 2. A Arquitetura do Webhook (A Sincronização Perfeita)

Usar um Webhook garante que a sua nuvem privada (o servidor onde o Akasha roda) nunca abra as portas para o GitHub; é o GitHub que bate na porta do seu servidor para avisar que há conhecimento novo.

O fluxo perfeito opera nesta sequência:

1. **O Gatilho Sagrado:** Você escreve um novo ritual ou expande a lenda de um Odu em Markdown no seu computador e faz o _merge_ na branch `main`.
    
2. **O Chamado (Webhook):** O GitHub dispara um payload (um aviso via POST) para uma rota oculta e autenticada no seu servidor (ex: `api/admin/webhooks/grimoire-sync`).
    
3. **A Verificação:** O seu backend recebe o chamado, verifica a assinatura criptográfica do GitHub (para garantir que não é um ataque) e aceita a ordem.
    
4. **O Motor Local (A Opção 3 em ação):** O próprio servidor executa o script interno (`npm run grimoire:sync`). Este script faz um `git pull` silencioso para puxar os novos arquivos Markdown.
    
5. **A Alquimia:** Como o script está rodando _dentro_ do servidor privado, ele se conecta via `localhost` ao Ollama, gera os novos _embeddings_ e os injeta de forma segura no `pgvector`.
    

Nenhum dado sensível trafega pela internet pública. Nenhuma senha de banco de dados sai do seu servidor.

Se por algum motivo o webhook falhar ou você precisar reindexar todo o banco após uma atualização no modelo do Ollama, você aperta o botão e o servidor refaz o trabalho.
